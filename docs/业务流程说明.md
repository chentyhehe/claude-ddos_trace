# DDoS 攻击溯源分析器 — 业务流程说明

## 一、系统定位

本系统是 DDoS 攻击发生**事后**的溯源分析工具，不是实时检测系统。

**输入**：一次 DDoS 攻击的告警 ID（或攻击目标 IP + 时间范围）

**输出**：攻击源 IP 列表、攻击团伙聚类、攻击入口路径、结构化报告

**数据来源**：ClickHouse 中的两张分布式表：
- `analytics_netflow_dist` — NetFlow 原始流量数据（攻击发生时的全网流量记录）
- `detect_attack_dist` — 攻击告警表（告警系统已检测到的攻击事件）

---

## 二、为什么需要这个系统

告警系统（`detect_attack_dist`）只告诉你**被谁打了**（攻击目标）和**什么时候打的**（时间窗口），但不告诉你：

| 告警系统已有的 | 本系统补充的 |
|-------------|-----------|
| 攻击目标 IP | 攻击来自哪些源 IP |
| 攻击类型（syn/udp 等） | 攻击源的行为指纹特征 |
| 峰值 PPS/BPS | 哪些攻击源是同一僵尸网络 |
| 阈值和持续时间 | 攻击从哪个骨干接口进入 |

简单说：**告警系统发现问题，溯源系统定位嫌疑人。**

## 四、各阶段详细说明

### Phase 0.5 — 告警上下文加载

**模块**：`data/alert_loader.py`

**输入**：`attack_id`（字符串，如 `"2026041417444901270003"`）

**数据源**：`detect_attack_dist` 表

**输出**：`AttackContext` 对象，包含以下字段：

| AttackContext 字段 | 来源列 | 含义 |
|-------------------|--------|------|
| target_ips | attack_target（type=ipv4） | 攻击目标 IP 列表 |
| target_mo_codes | attack_target（type=mo） | 监测对象编码列表 |
| start_time / end_time | 同名字段 | 攻击时间窗口（多条告警取最宽范围） |
| threshold_pps / threshold_bps | threshold + threshold_unit | 告警检测阈值（自动单位转换） |
| attack_types | attack_types（逗号分隔） | 攻击类型列表（合并去重） |
| max_pps / max_bps | 同名字段 | 告警期间峰值统计（取最大值） |

**阈值解析逻辑（三层 fallback）**：

```
告警表 threshold + threshold_unit
  ├─ 有值 → 自动单位转换 (Mbps→×10^6, Gbps→×10^9, Kbps→×10^3)
  └─ 无值 → 告警表 max_pps × 0.5 反推
              ├─ 有值 → 取峰值一半作为保守阈值
              └─ 无值 → threshold_pps / threshold_bps 保持 None
                         └─ get_pps_threshold() fallback 到配置默认值 (500K PPS / 20M BPS)
```

**为什么需要从告警表获取阈值**：不同攻击目标的基线差异巨大。IDC 服务器的 PPS 基线可能是 50 万，小型企业网关可能只有 5000。从告警表获取阈值是最合理的方式。

#### Phase 0.5+ — MySQL 多攻击类型阈值加载 + 攻击类型定义

**模块**：`data/mysql_loader.py`（`ThresholdLoader` 类）

**设计背景**：告警表中的阈值只代表触发告警的单一攻击类型阈值。但一次攻击往往涵盖多种攻击类型（如 SYN Flood + UDP Flood），每种类型的触发阈值各不相同。为提高异常检测精度，需要从 MySQL 业务数据库中按监测对象加载各攻击类型的独立阈值。

**本阶段输出的两个关键数据**：

1. **攻击类型定义**（`attack_type_info`）：47 种攻击类型的匹配规则（协议、端口、TCP 标志等），用于 Phase 0+ 的 Flow 分类
2. **每类型独立阈值**（`attack_thresholds`）：每种攻击类型的 PPS/BPS 触发阈值，用于 Phase 1.5 的基线构建和 Phase 2 的贡献度计算

**数据源**：MySQL 的 4 张配置表

| 查询步骤 | 表名 | 查询条件 | 获取信息 |
|----------|------|---------|---------|
| 1 | `system_base_monitor_object` | `WHERE code = ?` | 监测对象信息 + `threshold_id` |
| 2 | `system_base_threshold_summary` | `WHERE id = ?` | 阈值模板确认 |
| 3 | `system_base_threshold_item` | `WHERE threshold_id = ?` | 各攻击类型的 PPS/BPS 阈值 |
| 4 | `system_base_attack_type` | 全表 | 攻击类型定义（匹配规则 + 名称映射） |

**输出**：`MonitorThreshold` 对象

```
MonitorThreshold
├── mo_code: "12345"
├── mo_name: "XX数据中心"
├── threshold_enable: True
├── attack_thresholds: Dict[str, AttackTypeThreshold]   ← 每类型独立阈值
│   ├── "TCP SYN Flood" → {ipv4: {pps_trigger: 1M, bps_trigger: 500M}, ...}
│   ├── "UDP Flood"     → {ipv4: {pps_trigger: 2M, bps_trigger: 1G},   ...}
│   └── ...
└── attack_type_info: Dict[str, AttackTypeInfo]         ← 匹配规则定义
    ├── "TCP SYN Flood" → {protocol_num: 6, tcp_flags: "SYN", ...}
    ├── "UDP Flood"     → {protocol_num: 17, ...}
    └── ...
```

**IP 版本自动检测**：根据目标 IP 地址判断使用 IPv4 还是 IPv6 维度的阈值。

**降级策略**（两层）：
1. **MySQL 不可用**：攻击类型定义从 `docs/source/system_base_attack_type.csv` 加载（47 种攻击类型的静态文件），阈值使用配置文件默认值
2. **MySQL + CSV 均不可用**：阈值使用配置文件默认值，无法按攻击类型分类 Flow，退化为统一分析模式

### Phase 0 — 数据加载与过滤

**模块**：`data/loader.py`

**输入**：
- `target_ips`：目的 IP 列表（来自 AttackContext）
- `target_mo_codes`：监测对象编码列表（来自 AttackContext）
- `start_time` / `end_time`：时间范围（来自 AttackContext）

**数据源**：`analytics_netflow_dist` 表

**输出**：NetFlow 原始 DataFrame（约 13 万条流记录 × 27 个字段）

**查询条件**：
- `dst_ip_addr IN (目标IP列表)` — 目的 IP 过滤
- `dst_mo_code IN (监测对象编码)` — 监测对象过滤
- `app_rcv_time BETWEEN start AND end` — 时间范围（使用分区键做分区裁剪）

**为什么用 `app_rcv_time` 而不是 `parser_rcv_time`**：
`app_rcv_time` 是分区键（`PARTITION BY toYYYYMMDD(app_rcv_time)`），ClickHouse 自动裁剪不相关分区；`parser_rcv_time` 只是普通列，用它过滤会全表扫描。

**预处理**（`DataPreprocessor.process`）：毫秒时间戳 → datetime 类型转换

### Phase 0+ — Flow 按攻击类型分类

**模块**：`data/mysql_loader.py`（`filter_flows_by_attack_type()` 函数）

**设计背景**：一次 DDoS 攻击可能同时包含多种攻击类型。不同攻击类型的流量特征、阈值和行为模式完全不同。系统先用攻击类型定义的匹配规则将原始 Flow 分类到各攻击类型的子集，然后对每种类型独立运行完整流水线。

**输入**：
- 全量 NetFlow DataFrame（Phase 0 输出）
- 触发的攻击类型列表（AttackContext.attack_types）
- 攻击类型定义（MonitorThreshold.attack_type_info）

**输出**：`{attack_type_name: filtered_df}` — 每种攻击类型的 Flow 子集

**分类逻辑**：

对每种触发的攻击类型，使用其 `AttackTypeInfo` 中的匹配规则过滤：

| 条件 | 匹配规则 | 空值处理 |
|------|---------|---------|
| protocol_num > 0 | `df.protocol == protocol_num` | 0或空表示"不限协议" |
| port_type="SRC" + port_list | `src_port.isin(ports)` | 空值表示"不限端口" |
| port_type="DST" + port_list | `dst_port.isin(ports)` | 空值表示"不限端口" |
| tcp_flags 非空 | `(tcp_flags & required_bits) == required_bits` | 空值表示"不限标志" |
| src_equal_dst="EQUAL" | `src_ip_addr == dst_ip_addr` | 非 EQUAL 则不限制 |

所有条件为 **AND** 关系，空值条件视为"不限制"（跳过）。全向量化 pandas 操作。

**分类特性**：
- 同一条 Flow 可能匹配多种攻击类型（如 TCP SYN 包既属于 "TCP SYN Flood" 也属于 "TCP Traffic"）
- 不匹配任何触发类型的 Flow 在总览中作为"未分类流量"统计
- 若某种攻击类型的 Flow 数 < 10，跳过该类型的分析

### Phase 1 — 特征工程（按攻击类型分别执行）

**模块**：`features/extraction.py`

**输入**：某种攻击类型过滤后的 Flow 子集

**输出**：~N 个唯一源 IP × 30+ 维特征（每行一个源 IP，索引为 `src_ip_addr`）

**注意**：每种攻击类型独立提取特征。同一 IP 在不同攻击类型下会有不同的特征值（因为参与聚合的 Flow 子集不同）。

**为什么按源 IP 聚合**：NetFlow 每条记录是一个"流"，一次 DDoS 攻击产生几十万条流。直接分析粒度太细，无法判断"这个 IP 是不是攻击源"。按源 IP 聚合后，每个 IP 的行为特征一目了然。

**四类特征的计算方式和用途**：

| 类别 | 特征 | 计算方式 | 用于什么 |
|------|------|---------|---------|
| **基础统计** | total_packets | sum(packets) | 量化流量规模 |
| | total_bytes | sum(octets) | 量化流量体积 |
| | avg/std/max/min_packets | groupby.agg | 后续计算 burst_ratio 等衍生指标的基础 |
| | dst_port_count | nunique(dst_port) | 行为模式判断：单端口攻击 vs 多端口扫描 |
| | protocol_count | nunique(protocol) | 行为模式判断：单协议 vs 多协议 |
| **衍生指标** | packets_per_sec (PPS) | total_packets / flow_duration | 核心检测指标：包速率 |
| | bytes_per_sec (BPS) | total_bytes / flow_duration | 核心检测指标：字节速率 |
| | bytes_per_packet | total_bytes / total_packets | 区分攻击类型：小包(SYN)/大包(反射) |
| | burst_ratio | max_packets / avg_packets | 突发性：正常<3，攻击>5 |
| **时序特征** | flow_interval_mean | 相邻流间隔均值(毫秒→秒) | 发送节奏：攻击工具间隔小且稳定 |
| | flow_interval_cv | std(interval) / mean(interval) | 规律性：CV≈0 = 攻击工具 |
| | burst_count | 间隔<1秒的次数 | 攻击密集程度 |
| | max_burst_size | 连续突发的最大长度(游程编码) | 攻击工具一次脉冲的规模 |
| **分类特征** | country/province/isp | groupby 众数 | 地理溯源 |
| | dominant_protocol | protocol 众数 | 攻击类型推断 |
| | dominant_dst_port | dst_port 众数 | 攻击目标分析 |

**全向量化**：所有计算使用 pandas groupby + agg，134K 条流记录聚合为数千个 IP 用时不到 1 秒。

### Phase 1.5 — 基线建模（按攻击类型分别构建）

**模块**：`detection/anomaly.py` → `TrafficBaseline`

**输入**：某种攻击类型的源 IP 特征（Phase 1 对该类型 Flow 子集的输出）

**输出**：两类阈值 + 基线统计量

| 输出项 | 含义 | 后续用途 |
|--------|------|---------|
| `effective_thresholds["packets_per_sec"]` | 自适应单源 PPS 阈值（该类型 Flow 的 P95 × 1.5） | PPS 因子的 adaptive_score |
| `effective_thresholds["bytes_per_sec"]` | 自适应单源 BPS 阈值（该类型 Flow 的 P95 × 1.5） | BPS 因子的 adaptive_score |
| `effective_thresholds["alert_pps_threshold"]` | **该攻击类型的 MySQL 静态 PPS 阈值** | 攻击贡献度分母 |
| `effective_thresholds["alert_bps_threshold"]` | **该攻击类型的 MySQL 静态 BPS 阈值** | 攻击贡献度分母 |
| `baseline_stats` | 6个指标的均值/中位数/标准差/P75~P99 | z-score 计算的参考基线 |

#### 计算原理（三步走）

**第一步：过滤疑似攻击 IP（排除干扰）**

```
对该攻击类型的 Flow 子集计算 PPS 和 BPS 的均值和标准差
任何 IP 满足 PPS > 均值+3σ 或 BPS > 均值+3σ → 排除
剩余 IP 视为"疑似正常"流量
如果剩余 < 10 个 → 退化使用全量数据
```

为什么用 3σ：正态分布中 3σ 覆盖 99.7%，超出即为极端异常。这些 IP 几乎可以确定是攻击源，不应参与"正常基线"的计算。

为什么用 OR 而不是 AND：分散型 DDoS 中，攻击源可能只在 PPS 维度异常（小包洪泛），或只在 BPS 维度异常（反射放大）。AND 逻辑会让这些 IP 混入基线，拉偏统计量。

**第二步：计算正常流量的统计量**

对过滤后的"正常"IP，计算 6 个核心指标的统计分布：

```
for 每个指标 (PPS, BPS, BPP, burst_ratio, interval_mean, interval_cv):
    series = 正常IP的该指标值
    计算: mean, median, std, P75, P90, P95, P99, max, min
```

**第三步：计算两类有效阈值**

```
自适应单源阈值（基于该类型 Flow 子集的分布）:
    PPS 阈值 = 正常 PPS_P95 × 1.5
    BPS 阈值 = 正常 BPS_P95 × 1.5
    安全下界 = max(P99, P95×2)  ← 防止阈值过低

静态阈值（来自 MySQL 该攻击类型的配置）:
    alert_pps = threshold_item 中该类型的 ipv4_pps_trigger_rate
    alert_bps = threshold_item 中该类型的 ipv4_bps_trigger_rate
    → 作为因子6（攻击贡献度）的分母
    → 例如 SYN Flood: PPS阈值=1M, BPS阈值=500M
    → 例如 UDP Flood: PPS阈值=2M, BPS阈值=1G
    MySQL 不可用时使用配置文件默认值
```
       但对 UDP Flood 只贡献 2%（低分），取最大值更准确
```

#### 为什么用 P95 × 1.5 而不是告警阈值

**核心问题**：告警表的阈值（如 100M bps）是"所有源 IP 加起来"的聚合流量阈值——"总流量超过这个值才告警"。但我们需要逐 IP 判断"这个 IP 是不是攻击者"。

**举例说明**：

```
场景: 告警阈值 100M bps, 100 个正常 IP 各 1M bps, 1 个攻击 IP 50M bps

方案A — 直接用告警阈值 (旧方案):
    检测阈值 = 100M → 50M 的攻击 IP 未达到 → 漏检!

方案B — 用 P95 × 1.5 (当前方案):
    正常 IP 的 BPS P95 ≈ 1M
    自适应阈值 = 1M × 1.5 = 1.5M
    攻击 IP 的 50M >> 1.5M → 检出!
```

### Phase 2 — 异常源检测

**模块**：`detection/anomaly.py` → `AnomalyDetector`

**输入**：
- Phase 1 输出的 3K 个源 IP 特征
- Phase 1.5 输出的两类阈值
- Phase 1.5 输出的基线统计量

**输出**：每个源 IP 新增 3 列：
- `attack_confidence`：攻击置信度（0-100）
- `traffic_class`：四级分类
- `confidence_reasons`：置信度说明文本

#### 6 因子评分模型

```
综合置信度 = 0.25×f1 + 0.20×f2 + 0.15×f3 + 0.15×f4 + 0.10×f5 + 0.15×f6
```

#### 各因子的详细计算原理

##### 因子1: PPS 相对偏离 (权重 25%)

```
输入: 每个 IP 的 packets_per_sec, 基线均值/标准差, 自适应PPS阈值

计算:
    z_score = clip((PPS - 基线均值) / 基线标准差 × 10, 0, 100)
    adaptive_score = PPS >= 自适应阈值 ? clip(PPS/自适应阈值 × 50, 50, 100) : 0
    f1 = max(z_score, adaptive_score)
```

设计意图：双重评分机制，任一维度判为异常都给高分。
- z_score 适合检测"相对同侪偏离大"的 IP
- adaptive_score 适合检测"绝对值超过正常 P95"的 IP

权重 25% 的理由：PPS 是 DDoS 最直接的指标，但不应过高（旧方案 30%），因为还有其他维度补充。

##### 因子2: BPS 相对偏离 (权重 20%)

```
计算逻辑与因子1完全相同，只是换成 BPS 维度
```

权重 20%（低于 PPS）的理由：大多数 DDoS 攻击以包速率（PPS）为首要特征。但 DNS 放大攻击等是 BPS 高但 PPS 不高，所以 BPS 仍需独立权重。

##### 因子3: 包大小异常 (权重 15%)

```
输入: 每个 IP 的 bytes_per_packet, 基线均值/标准差

计算:
    z_abs = |bytes_per_packet - 基线均值| / 基线标准差
    f3 = clip(z_abs × 15, 0, 100)
```

设计意图：**双侧检测**——小包（SYN Flood ~40-60B）和大包（反射放大 >1400B）都异常。
缩放因子 15 使得偏离约 6.7 个标准差时达到满分。

##### 因子4: 突发模式 (权重 15%)

```
输入: burst_ratio, burst_count, max_burst_size

计算:
    ratio_score = clip(burst_ratio / 5.0 × 100, 0, 100)
    count_score = clip(burst_count / 100 × 100, 0, 100)
    size_score  = clip(max_burst_size / 20 × 100, 0, 100)
    f4 = 0.4 × ratio_score + 0.3 × count_score + 0.3 × size_score
```

设计意图：攻击工具通常产生规律性的突发流量。三个子维度覆盖不同类型的突发：
- burst_ratio：整体突发程度（最大/平均）
- burst_count：突发频率
- max_burst_size：单次突发规模

##### 因子5: 行为模式 (权重 10%)

```
输入: dst_port_count, protocol_count, flow_interval_cv, flow_count

计算:
    single_port = dst_port_count == 1
    single_proto = protocol_count == 1
    regular = (flow_interval_cv < 0.5) 且 (flow_count >= 5)
    f5 = (条件满足数 / 3) × 100
    满足0个=0分, 1个=33分, 2个=67分, 3个=100分
```

设计意图：单端口 + 单协议 + 规律发送 = 典型攻击工具特征。权重较低(10%)是因为某些正常流量（如单一 CDN 节点回源）也可能满足部分条件，不宜给过高权重。

##### 因子6: 攻击贡献度 (权重 15%)

```
输入: 每个 IP 的 PPS/BPS, 告警聚合阈值或全量总和

分母选择:
    告警阈值有效 (非fallback默认值) → 分母 = 告警阈值
    告警阈值是 fallback 默认值 → 分母 = 全量数据的 PPS/BPS 总和

计算:
    contribution = max(IP_PPS/分母PPS, IP_BPS/分母BPS)
    f6 = 分段评分:
        < 1%    → 0 分   (贡献极微)
        1%-5%   → 0-40   (有一定贡献)
        5%-20%  → 40-70  (显著贡献)
        ≥ 20%   → 70-100 (核心攻击源)
```

设计意图：在分散型 DDoS 中（数千个僵尸节点，每个只有几千 PPS），单个 IP 的 z-score 可能不高，但如果它贡献了总流量的 5%+，仍然是核心攻击源。

**分母 fallback 策略的重要性**：当告警表缺少阈值时，旧方案用 fallback 默认值 500K PPS 做分母，导致每个 IP 贡献度 < 1%（f6=0）。改为用全量总和做分母后，贡献度能正确反映"该 IP 占实际攻击总量的比例"。

#### 四级分类

| 分类 | 置信度 | 含义 | 操作建议 |
|------|--------|------|---------|
| confirmed | ≥ 80 | 6因子综合判断为攻击源 | 建议封禁 |
| suspicious | 60-80 | 高度可疑 | 建议持续监控 |
| borderline | 40-60 | 无法确定 | 保留观察 |
| background | < 40 | 正常流量 | 无需处理 |

#### 权重设计的合理性分析

**总权重 = 25+20+15+15+10+15 = 100%，6 个因子各自独立。**

| 因子 | 权重 | 合理性 | 潜在问题 |
|------|------|--------|---------|
| PPS偏离 | 25% | 合理。PPS 是 DDoS 最直接的指标 | 分散攻击中 z-score 可能被压缩 |
| BPS偏离 | 20% | 合理。补充 DNS 放大等 PPS 不高的攻击 | 与 PPS 有一定相关性 |
| 包大小 | 15% | 合理。区分攻击类型的重要特征 | 正常 VPN 流量也可能有异常包大小 |
| 突发模式 | 15% | 合理。攻击工具的典型特征 | CDN 突发流量可能误判 |
| 行为模式 | 10% | 偏低但合理。权重低是因为正常流量也可能满足部分条件 | 可能需要增加"固定间隔"的权重 |
| 攻击贡献度 | 15% | 合理。是分散型攻击的关键检测维度 | 依赖准确的分母（阈值或总和） |

**关键阈值调节建议**：

| 参数 | 当前值 | 影响范围 | 调节方向 |
|------|--------|---------|---------|
| P95_BUFFER_FACTOR | 1.5 | 自适应阈值高低 | 降低→更敏感（多检出但可能误报）；升高→更保守 |
| 确认阈值 | 80 | confirmed 数量 | 降低→更多 IP 被确认；升高→更严格 |
| 可疑阈值 | 60 | suspicious 数量 | 降低→更多 IP 被标可疑 |
| 边界阈值 | 40 | borderline 数量 | 这是大多数 IP 的分界线，一般不需调整 |
| 贡献度分段 | 1%/5%/20% | f6 得分分布 | 降低分段→更敏感 |

### Phase 3 — 指纹聚类

**模块**：`clustering/fingerprint.py`

**输入**：Phase 2 筛选的 confirmed + suspicious 异常源（含 30+ 维特征）

**输出**：聚类报告 DataFrame（每个攻击团伙一行），可能为 None（异常源不足时跳过）

**聚类报告字段**：

| 字段 | 含义 |
|------|------|
| cluster_id | 团伙编号 |
| member_count | 成员 IP 数量 |
| member_ips | 成员 IP 列表（逗号分隔） |
| avg_bytes_per_packet 等 9 个 avg_ 字段 | 团伙的平均行为指纹 |
| attack_type | 推断的攻击类型 |

**9 维聚类特征**：

| 特征 | 为什么能区分团伙 |
|------|---------------|
| bytes_per_packet | 不同攻击工具的包大小不同 |
| packets_per_sec, bytes_per_sec | 不同工具的速率特征不同 |
| burst_ratio, burst_count | 突发模式反映工具的发送策略 |
| flow_interval_mean, flow_interval_cv | 发送间隔反映工具的定时器精度 |
| dst_port_count, protocol_count | 攻击目标端口和协议的多样性 |

**三层算法降级**：

```
HDBSCAN（首选）→ DBSCAN（次选）→ MiniBatchKMeans（兜底）
```

| 算法 | 优势 | 劣势 | 降级触发条件 |
|------|------|------|------------|
| HDBSCAN | 无需预设簇数，自动发现任意形状簇 | 需额外安装 | 未安装或运行失败 |
| DBSCAN | sklearn 内置，密度聚类 | 需手动设 eps | 运行失败 |
| MiniBatchKMeans | 计算最快 | 需预设簇数 | 始终能运行 |

**攻击类型推断规则（按优先级）**：
1. BPP < 100 且 TCP(6) → SYN Flood
2. BPP < 100 → 小包洪泛
3. BPP > 1400 → 大包洪泛（反射放大）
4. Protocol = 17 → UDP Flood
5. Protocol = 1 → ICMP Flood
6. Protocol = 6 → TCP Flood
7. 其他 → 混合型攻击

## 六、关键设计决策总结

| 决策 | 选择 | 原因 |
|------|------|------|
| **主流程** | 按攻击类型分项分析 | 不同攻击类型的流量特征和阈值完全不同，统一分析会互相干扰 |
| 阈值来源 | MySQL 按攻击类型独立阈值 | 不同攻击类型的阈值各不相同，按类型分别计算贡献度更精确 |
| 攻击类型定义 | MySQL 优先 + CSV 降级 | MySQL 包含最新定义，CSV 保证 MySQL 不可用时仍能分类 Flow |
| 基线过滤 | 3σ OR 逻辑排除疑似攻击 IP | 防止分散型攻击源混入基线拉偏统计量 |
| 单源阈值 | 正常 P95 × 1.5（自适应） | 告警阈值是聚合维度的，不应直接当单源阈值 |
| 攻击贡献度分母 | 该攻击类型的 MySQL 静态阈值 | 直接用该类型的触发阈值做分母，最准确反映贡献度 |
| MySQL 容错 | 不可用时回退到 CSV + 配置默认值 | MySQL 是增强数据源，不是必需数据源 |
| 评分模型 | 6 因子加权 | 综合多维度信息，降低误报和漏报 |
| 聚类算法 | 三层降级 | 兼顾精度和鲁棒性 |
| 特征计算 | 全向量化 pandas | 13 万条记录在 1 秒内完成聚合 |

## 八、已知局限与改进方向

| 局限 | 影响 | 改进方向 |
|------|------|---------|
| 告警表阈值缺失时依赖 fallback 默认值 | 贡献度分母不准 | 用全量总和 fallback（已实现） |
| 分散型攻击的 3σ 过滤可能不够 | 攻击源混入基线 | 迭代过滤或 MAD 替代 std |
| z-score 对非正态分布不理想 | 评分偏差 | 考虑百分位排名替代 z-score |
| 权重固定不可调 | 不同场景不够灵活 | 支持通过配置文件调整权重 |
| 不积累历史情报 | 每次分析从零开始 | 构建威胁情报库（见第九章） |
