# DDoS 攻击溯源分析器功能介绍

## 1. 项目定位

本项目是一个基于 NetFlow 数据的 DDoS 攻击溯源分析工具，面向已经发生或正在发生的攻击事件，帮助分析人员回答以下问题：

- 攻击目标是谁
- 攻击来自哪些源 IP
- 哪些源 IP 可以判定为核心攻击源
- 多个攻击源之间是否属于同一攻击团伙
- 攻击流量从哪些入口设备或接口进入网络
- 攻击源主要来自哪些国家、地区、ISP 或监测对象

系统并不是实时检测引擎，而是建立在现有告警与流量数据之上的分析器。它通过读取 ClickHouse 中的告警表和 NetFlow 明细表，结合 MySQL 中维护的攻击类型定义与阈值配置，自动生成结构化分析结果。

## 2. 解决的问题

告警系统通常只能告诉使用者某个目标在某个时间窗口内遭受了 DDoS 攻击，但无法进一步回答攻击源、攻击团伙和攻击入口等问题。本项目补齐了这些分析环节，提供：

- 告警驱动的一键分析能力
- 按攻击类型拆分的精细化分析能力
- 基于统计基线的异常源识别能力
- 基于行为指纹的攻击团伙聚类能力
- 基于地理和入口信息的路径重构能力
- 面向控制台、CSV 和图表的报告输出能力

## 3. 数据来源

项目当前主要依赖三类数据源：

### 3.1 ClickHouse NetFlow 数据

- 表名默认是 `analytics_netflow_dist`
- 用于读取原始流量记录
- 关键字段包括源/目的 IP、包数、字节数、协议、端口、TCP 标志、入口接口、采集设备 IP、地理归属和监测对象信息

### 3.2 ClickHouse 攻击告警数据

- 表名默认是 `detect_attack_dist`
- 用于按 `attack_id` 或 `attack_target` 获取告警上下文
- 可自动提取目标 IP、监测对象、时间窗口、攻击类型、阈值、峰值流量等信息

### 3.3 MySQL 阈值与攻击类型配置

- 读取监测对象配置、阈值模板、阈值明细、攻击类型定义
- 为每种攻击类型提供独立的 PPS/BPS 阈值
- 为流量分类提供协议、端口、TCP 标志等匹配规则

### 3.4 CSV 降级数据源

当 MySQL 不可用时，系统会从 `docs/source/system_base_attack_type.csv` 加载攻击类型定义，保证“按攻击类型筛流”这一步仍可运行。此时每种攻击类型的阈值会退化为配置文件中的默认阈值。

## 4. 输入方式

项目支持三种主要分析入口。

### 4.1 基于告警 ID 分析

这是推荐入口。

- 输入一个 `attack_id`
- 系统自动查询告警记录
- 自动拼出分析所需的目标、时间窗口和攻击类型
- 自动加载监测对象对应的攻击类型阈值

适合对接现有告警平台，在告警触发后直接执行溯源。

### 4.2 基于攻击目标分析

- 输入攻击目标 `attack_target`
- 可选传入开始和结束时间
- 系统会先尝试从告警表中找到相关记录
- 如果找到多条，会按不同告警上下文分别分析
- 如果找不到，则退化为最小上下文，继续用默认阈值分析

适合按单个 IP、监测对象编码或业务目标做追查。

### 4.3 手动参数分析

- 手动指定目标 IP 列表
- 手动指定目标监测对象列表
- 手动指定开始和结束时间

该模式不依赖告警上下文，主要用于调试、补分析或兼容旧接口。

## 5. 核心功能

### 5.1 告警上下文自动构建

系统会从告警表中构建 `AttackContext`，整合出：

- 攻击目标
- 目标类型
- 目标 IP 列表
- 监测对象编码列表
- 开始和结束时间
- 攻击类型列表
- 告警级别
- 最大 PPS/BPS
- 检测阈值

对于同一个 `attack_id` 的多条告警，系统会自动合并：

- 取最早开始时间
- 取最晚结束时间
- 合并全部攻击类型
- 取最大阈值
- 从 `custcode` 提取监测对象编码

同时支持将 `pps`、`kpps`、`mpps`、`bps`、`Kbps`、`Mbps`、`Gbps` 等单位统一换算为标准值。

### 5.2 原始流量加载与预处理

系统从 ClickHouse 查询 NetFlow 明细后，会执行预处理：

- 将 `parser_rcv_time` 转成 `flow_time`
- 将 `first_time`、`last_time` 转成可读时间
- 修正包数、字节数、端口、协议、接口等字段的数据类型

当前查询支持按：

- 目的 IP 过滤
- 目的监测对象过滤
- 时间范围过滤

查询结果会按 `parser_rcv_time` 排序，便于后续时序分析。

### 5.3 按攻击类型拆分流量

这是当前项目的核心设计之一。

系统会先根据告警触发的攻击类型，读取相应攻击定义，再将原始 NetFlow 过滤成多个攻击类型子集。匹配规则来自 MySQL 或 CSV 中的攻击类型定义，主要包括：

- 协议号匹配
- 源端口或目的端口匹配
- TCP 标志位匹配
- 源 IP 是否等于目的 IP

这样做的目的是让不同攻击类型各自运行完整分析流程，避免把 TCP SYN Flood、UDP Flood、ICMP Flood 这类差异很大的流量混在一起建模。

### 5.4 特征工程

系统按源 IP 聚合，提取多维行为特征，主要包括以下几类。

#### 基础统计特征

- 总包数、平均包数、标准差、最大值、最小值
- 总字节数、平均字节数、标准差、最大值
- 目标端口数、源端口数、协议数
- 入口接口数、出口接口数、采集设备数
- 流开始时间、结束时间、持续时间、流数

#### 衍生指标

- `bytes_per_packet`
- `packets_per_sec`
- `bytes_per_sec`
- `burst_ratio`
- `bytes_std_ratio`

#### 时序特征

- 相邻流量时间间隔均值
- 时间间隔标准差
- 时间间隔变异系数
- 突发次数
- 最大连续突发长度
- 活跃度占比

#### 分类和标签特征

- 国家、省份、城市、ISP 众数
- AS 号众数
- 源和目的监测对象众数
- 主导协议
- 主导目的端口
- 主导 TCP 标志

### 5.5 正常流量基线建模

系统不是直接拿固定阈值去判断单个 IP，而是会先尝试构建“正常流量基线”，再衡量每个 IP 相对同类流量的偏离程度。

基线构建流程包括：

1. 先从所有源 IP 中剔除明显异常的流量源
2. 再对剩余 IP 计算关键指标的均值、中位数、标准差、P75、P90、P95、P99
3. 基于这些统计量计算自适应阈值

异常剔除支持三层策略：

- `sigma`：均值加标准差方式
- `median + MAD`：更稳健的鲁棒过滤
- `percentile`：按排名排除 Top 百分比

如果排除后正常样本太少，系统会自动退化为使用全量数据，避免基线失真到无法计算。

### 5.6 多因子异常检测

每个源 IP 会被计算一个 0 到 100 的攻击置信度。当前实现使用 6 个因子加权得分：

- PPS 相对偏离
- BPS 相对偏离
- 包大小异常
- 突发模式
- 行为模式
- 攻击贡献度

分类结果分为四个等级：

- `confirmed`
- `suspicious`
- `borderline`
- `background`

除了最终得分，系统还会生成可解释的原因文本，比如：

- PPS 相对偏离得分高
- BPS 相对偏离得分高
- 包大小异常
- 突发模式明显
- 行为模式像自动化工具
- 攻击贡献度高

### 5.7 攻击团伙聚类

对于 `confirmed` 和 `suspicious` 的异常源，系统会继续做攻击指纹聚类，用来识别可能属于同一僵尸网络或攻击工具的一组源 IP。

聚类使用 9 维特征：

- bytes_per_packet
- packets_per_sec
- bytes_per_sec
- burst_ratio
- burst_count
- flow_interval_mean
- flow_interval_cv
- dst_port_count
- protocol_count

聚类算法采用三级降级策略：

1. HDBSCAN
2. DBSCAN
3. MiniBatchKMeans

当样本过大时，系统会先采样训练，再通过 1-NN 回传标签，以降低内存压力。

每个聚类结果还会自动推断一个简化攻击类型，例如：

- SYN Flood
- UDP Flood
- ICMP Flood
- 小包洪泛
- 大包洪泛
- 混合型攻击

### 5.8 攻击路径重构

路径重构只针对 `confirmed` 和 `suspicious` 的源 IP 进行，重点输出四类信息。

#### 入口路由器分析

按以下维度聚合：

- `flow_ip_addr`
- `input_if_index`

用于识别攻击流量进入网络的主要入口节点。

#### 地理来源分析

按以下字段聚合：

- `src_country`
- `src_province`
- `src_city`
- `src_isp`

用于识别攻击源的地域分布和运营商归属。

#### 监测对象关联分析

按以下字段聚合：

- `src_mo_code`
- `src_mo_name`

用于关联攻击流量来源于哪个监测对象或业务对象。

#### 时间分布分析

按小时聚合：

- 流数
- 活跃源 IP 数
- 总包数

用于观察攻击波次、峰值时段和持续情况。

### 5.9 文字报告与可视化输出

项目会生成结构化文字报告，内容通常包括：

- 分析时间窗口
- 源 IP 总量
- 分类摘要
- Top 攻击源
- 聚类摘要
- 来源地域摘要
- 来源监测对象摘要
- 峰值攻击时段摘要
- 按攻击类型分项的详细分析

同时还会导出 CSV、PNG、Markdown 和 JSON 文件，既便于人工查看，也便于自动化平台消费。

图形输出经过统一字体配置后，会优先使用系统中的中文字体，避免常见的中文乱码和负号显示异常问题。

### 5.10 跨攻击类型聚合策略

在改进后的设计里，系统不再简单地把不同攻击类型的结果按“首次出现”合并，而是采用“最高风险视图”聚合策略：

- 同一个源 IP 如果命中多个攻击类型，聚合层会优先保留攻击置信度最高的那一条记录
- 同时保留该 IP 命中的全部攻击类型列表
- 输出 `best_attack_type`
- 输出 `matched_attack_types`
- 输出 `matched_attack_type_count`
- 输出跨类型最高置信度

这样可以避免总览、黑名单和 API 摘要受到遍历顺序影响，更适合真实的多类型混合攻击场景。

## 6. 输出结果

系统当前会根据分析结果导出以下文件。

### 6.1 基础分类结果

- `traffic_classification_report{tag}.csv`
  - 记录每个源 IP 的分类、置信度、关键特征和地理信息

### 6.2 聚类结果

- `cluster_fingerprint_report{tag}.csv`
  - 每个聚类一行，包含成员数、成员 IP、平均指纹和推断攻击类型

- `cluster_radar_chart{tag}.png`
  - 按聚类绘制雷达图

### 6.3 威胁情报与联动输出

- `attack_blacklist{tag}.csv`
  - 输出确认和可疑攻击源，可用于封禁或联动 SOAR
  - 额外包含最佳攻击类型、命中的攻击类型列表、跨类型最高置信度等聚合字段

- `attack_timeline{tag}.csv`
  - 输出按小时的攻击时间线
  - 包含每个时段的 Top 攻击源和总字节量

- `analysis_summary{tag}.json`
  - 输出总览摘要、阈值摘要、路径摘要和各攻击类型摘要
  - 适合被外部系统直接读取

- `analysis_report{tag}.md`
  - 将控制台文字报告落盘，便于归档和分享

- `overall_attack_situation{tag}.md`
  - 输出总体攻击态势说明
  - 包括异常源占比、核心攻击源、主要来源地域、主要入口节点、峰值攻击时段等摘要信息

### 6.4 按攻击类型分项输出

- `attack_type_detail{tag}.csv`
  - 每行表示一个源 IP 在某种攻击类型下的分析结果
  - 除分类和置信度外，还会包含匹配规则、阈值、主导协议、主导端口、包大小、突发比等信息

- `suspicious_sources{tag}.csv`
  - 输出确认和可疑攻击源，按攻击类型展开，并附带可疑原因
  - 额外包含总包数、总字节数、平均包大小、突发比、流记录数等字段

- `attack_overview{tag}.png`
  - 生成总览仪表盘

- `attack_timeline_chart{tag}.png`
  - 输出攻击时间趋势图
  - 直观看出攻击包数和活跃源 IP 的时间变化

- `source_distribution_dashboard{tag}.png`
  - 输出来源分布仪表盘
  - 同时展示地域来源、来源监测对象和入口节点分布

- `top_attacker_radar_chart{tag}.png`
  - 输出核心攻击源行为雷达图
  - 用于直观看多个高风险源之间的行为差异

如果某个攻击类型本身也聚出了攻击团伙，还会额外生成按攻击类型命名的雷达图文件。

### 6.5 路径分析专项输出

- `entry_router_report{tag}.csv`
  - 输出入口设备和接口分布

- `geo_distribution_report{tag}.csv`
  - 输出攻击来源地域分布

- `mo_distribution_report{tag}.csv`
  - 输出来源监测对象分布

- `time_distribution_report{tag}.csv`
  - 输出按小时聚合的攻击时间分布

## 7. 对外接口

每次分析的全部结果会自动输出到独立目录中，目录名格式为：

- `攻击ID_目标IP`

例如：

- `ATK-20260401-001_192.168.1.100`
- `manual_192.168.1.100`

这样可以避免多次分析的结果混放在同一个目录中，便于归档、复盘和按攻击事件检索。

### 7.1 命令行接口

项目提供 `ddos-trace` 命令以及 `python -m ddos_trace` 方式，支持以下子命令：

- `serve`
- `alert`
- `target`
- `analyze`

分别对应：

- 启动 API 服务
- 基于告警 ID 分析
- 基于攻击目标分析
- 手动传参分析

### 7.2 HTTP API

项目内置 FastAPI 服务，提供以下接口：

- `GET /health`
- `POST /api/v1/analyze/alert`
- `POST /api/v1/analyze/target`
- `POST /api/v1/analyze`

返回内容包括：

- 任务 ID
- 分析状态
- 告警上下文
- 分类摘要
- 异常源列表
- 聚类列表
- 地理分布
- 入口路由器信息
- 各攻击类型摘要
- 完整文字报告

## 8. 配置能力

项目配置集中在根目录 `config.yaml`，主要包括：

- ClickHouse 连接
- MySQL 连接
- 默认阈值
- 聚类与基线参数
- API 监听地址
- 输出目录
- CSV 降级文件路径

配置优先级为：

- 环境变量
- `config.yaml`
- 代码默认值

这使得项目既适合本地调试，也适合在服务环境中通过环境变量覆盖配置。

## 9. 当前实现特点

结合当前代码，可以把项目的实现特点概括为：

- 以“按攻击类型分项分析”为主流程
- 以“告警驱动分析”为推荐入口
- 以“统计基线 + 多因子评分”作为核心识别策略
- 以“MySQL 优先、CSV 降级”作为配置容灾策略
- 以“最高风险视图聚合”作为跨攻击类型总览策略
- 以“文本报告 + CSV + 图表 + JSON”作为分层交付结果

## 10. 适用场景

本项目适合以下场景：

- 安全运营人员对已触发的 DDoS 告警做事后溯源
- 网络侧分析人员定位攻击源、入口路由器和来源区域
- 平台侧将分析结果导入黑名单、ACL 或自动化处置平台
- 开发和算法人员基于真实流量迭代异常检测与聚类策略

## 11. 使用建议

如果要在实际场景中发挥更好效果，建议优先使用以下模式：

- 优先走告警 ID 入口，减少人工录入参数
- 保持 MySQL 阈值配置可用，以便按攻击类型使用独立阈值
- 将输出目录与自动化处置链路打通，复用 `attack_blacklist`、`suspicious_sources`、`analysis_summary` 等结果
- 在不同网络环境下根据业务特征调整默认阈值和异常剔除策略

## 12. 本轮设计优化

相较于前一版，当前设计已经做了以下优化：

- 修正跨攻击类型聚合逻辑，避免同一源 IP 仅按首次出现保留结果
- 按攻击 ID 和目标 IP 自动创建独立输出目录
- 修复图形输出中的中文显示问题
- 丰富总览报告内容，增加地域、监测对象和时间分布摘要
- 增加文字报告落盘能力，避免结果只停留在控制台输出
- 增加 JSON 摘要输出，便于自动化系统消费
- 增加路径分析专项 CSV，便于网络侧和安全侧分别使用
- 增加总体攻击态势说明、趋势图和来源分布仪表盘
- 增强雷达图表现，补充核心攻击源雷达图
- 补充更多分项字段，使导出结果更适合研判、复盘和联动处置
